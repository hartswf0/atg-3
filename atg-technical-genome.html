ATG Architecture: The Technical Genome

A definitive guide to the "Gothic Assemblage" constructing the Trail Olog, Pattern Glossary, and Document Genome.

The ATG system is not a static website but a living knowledge graph. Unlike traditional CMS architectures that rely on databases, ATG uses a "Gothic" assemblage of JSON schemas, POML strategies, and HTML5 visualization engines. This document details exactly how the three core systems—Transformation (Olog), Reflection (Glossary), and Definition (Genome)—are technically constructed and interwoven.

1. The Trail Olog Map (`courage-trail-olog.html`)

The Trail Olog is a topological terrain map that visualizes the project's evolution. It does not use external mapping libraries; it is a custom-built SVG engine that renders a "10x8 grid topology" based on JSON data.

DATA SOURCE: courage-trail-data.json │ ▼ LOADER (JS): loadTrailData() │ • Fetches JSON │ • Fallback: embedded data (if CORS fails) │ • Parsed into TRAIL_DATA global object ▼ RENDER ENGINE: draw() (HTML5 Canvas + SVG Overlay) │ ├── DRAW ZONES (The "Islands") │ • Iterate TRAIL_DATA.zones │ • Map grid (col,row) → Pixel (x,y) │ • Draw SVG shapes based on type (mountain, forge, river) │ • Color code by Trail Level (0-3) │ ├── DRAW CONNECTIONS (The "Trails") │ • Iterate TRAIL_DATA.connections │ • Draw Bézier curves between Zone centers │ • Animate "Pilgrims" (white dots) following curves │ └── DRAW TIMELINE • Render bottom scrubber • Filter Zones by active date range

Technical Core: The 10x8 Grid Topology

The map abandons "pixel-perfect" design for a semantic grid. Every zone is defined by a `(col, row)` coordinate in a virtual 10x8 grid. This ensures that the layout is responsive and topologically consistent regardless of screen size. The `getPoint(col, row)` function translates these abstract coordinates into actual canvas pixels.

The "Pilgrim" Animation System

The moving dots ("Pilgrims") are not GIFs. They are procedural classes (`Pilgrim`) spawned by the `spawnPilgrims()` function. They:

  • Select a random connection from `TRAIL_DATA.connections`.
  • Traverse the Bézier curve using a progress variable `t` (0 to 1).
  • Upon reaching `t=1`, they die and respawn, simulating continuous data flow ("Data Exhaust").

2. The Pattern Glossary (`pattern-glossary.html`)

The Pattern Glossary acts as the self-reflective engine of the system. It validates the "implicit theory" behind the explicit code.

Audit Protocol (CSS Classes)

Academic rigor is enforced via CSS classes that visually distinguish claim strength:

.audit-verified (Green): Proven archival source.
.audit-deep (Purple): Philosophical/theoretical grounding.
.audit-edge (Blue/Green): Bleeding-edge (2024+) research.
.void-note (Red): Warnings/Absence of citations.
The "Pattern Card" Component

Each pattern is an <article> element containing:

  • Badge: Visual proof of audit status.
  • Zone Chips: Where it lives in the Olog map.
  • Definition: 1-2 sentence condensed theory.
  • Grounding: Direct link to primary source.

The "Sociotechnical" Verification

The glossary is not just a list of terms; it is a verified graph. It links design patterns (like "Context Engineering") to their academic ancestors ("Gothic Assemblage" by Turnbull). This is hard-coded in the HTML structure, creating a "frozen" snapshot of theoretical validity.

3. The Document Genome (`project-genome.html` & `genome.poml`)

The Genome is the schema definition layer. It uses "Ologs" (Ontology Logs) to enforce mathematical rigor on the project's structure.

SCHEMA SOURCE: genome.poml │ ▼ DEFINES: The 8 Codons (C-T-A-M-D-S-P-I) │ • C: Creator • D: Decision │ • T: Trail • S: Trail-Aware Tool │ • A: Artifact • P: Pattern │ • M: Metadata • I: Intent │ ▼ ENFORCES: The Morphisms (Arrows) • "generates" (Creator → Trail) • "observes" (Tool → Trail) • "anneals" (Pattern → Structure) │ ▼ VISUALIZES: project-genome.html • Renders the Olog Types as cards • Maps the Morphisms as logical flows

POML: The Prompt Object Markup Language

The Genome is written in POML (`genome.poml`). This is a specialized pseudo-code format designed for LLMs. It defines:

  • Roles: Who the agent is (e.g., "Void Guardian").
  • Protocols: Step-by-step algorithms for tasks (e.g., "Citation Triage").
  • Axioms: Unbreakable rules (e.g., "If you can't cite it, you can't claim it").

The `project-genome.html` page effectively "compiles" this POML logic into a human-readable visual guide.

4. Synthesis: How They Connect

These three systems form a closed loop of "Associative Infrastructuring":

1. DEFINE (Genome): genome.poml defines the rules of the universe (Creator, Trail, Artifact).

2. BUILD (Data): courage-trail-data.json instantiates those rules into specific zones and connections.

3. MAP (Olog): courage-trail-olog.html reads the data and renders the spatial reality.

4. REFLECT (Glossary): pattern-glossary.html observes the reality and validates the theory behind it.

This architecture moves beyond "software engineering" into "Ontological Engineering." We are not just building tools; we are building a verifiable, self-documenting knowledge system that knows why it exists.
