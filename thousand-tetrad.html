From the Failure of TILTH to a Design Pattern for Scenario Simulation

Part of LMC-6650 Values in Design · Launch Tool · 67 Scenarios · 12 Categories

The Failure of TILTH

TILTH Investigation The speculative company that taught void management

TILTH was a speculative company—a world inside of a world, a speculative investigation of a speculative company. The scenarios became convoluted quickly. It was hard to show enough of the world for it to gain coherency.

The project was stuck in the trough of quality: almost good enough to stand alone, but the hard parts of completion (final polish, coding refinements) got exponentially more challenging as the project context grew.

The core tension: Designing tools that design the world vs. designing for designers. TILTH was a chance to test my own tools and find opportunities to explore collaborations—a way of bootstrapping other projects.

Constraints That Mattered

  • Must work on a mobile phone
  • Make prompting and simulation exploration easier
  • Became a place to understand affordances

What TILTH Taught

The bigger question emerged: how to deal with many branching worlds? TILTH got harder to use each time I went back and added to it. I needed infrastructure that gets better and richer the more you use it—not worse.

The Genesis of Thousand-Tetrad

The impetus came from realizing that much better speculative worlds had already been created. Rather than building worlds, I could be parasitic off existing universes—focusing on novel affordances and the technical tool-building process instead of keeping track of the world itself.

Inspirations

  • Ted Chiang — stories that present ethical dilemmas in stunning detail
  • Life After BOB — AI ethics scenarios I had observed through working with simulation architects
  • 1000 Lives — Shuruq Tramontini's work on AI consciousness narratives
Goal: Improve and grow with simulations. Learn from both the technical project and the collaborators. Reach a technical threshold to be recognized as a future collaborator on such projects.

The Core Question

Build a design pattern that can be scaled and operationalized across any scenario—one that stays invariant regardless of the world being explored.

The Design Pattern

┌──────────────────────────────────────────────────────┐ │ [Corner] HEADER / OPERATORS [Corner] │ └──────────────────────────────────────────────────────┘ ┌──────────────────────────────────────────────────────┐ │ │ │ 9 × 9 G R I D │ │ │ │ Any entity can be controlled via grid or chat │ │ Anything can be branched │ │ │ └──────────────────────────────────────────────────────┘ ┌──────────────────────────────────────────────────────┐ │ C H A T │ │ Any new UI loads through here │ └──────────────────────────────────────────────────────┘ ┌──────────────────────────────────────────────────────┐ │ [Corner] Scene Selector │ Tetrad Op [Corner] │ └──────────────────────────────────────────────────────┘

This became the invariant structure: a 9×9 grid with a chat beneath it, a header with two corner buttons and operators in the middle, a footer with scene selector and tetrad operators. Any new UI loads through the chat. Any entity can be controlled via the chat or the grid. Anything can be branched.

The Discovery Moment

Nexadoc-Jar: The Origin Where prompt/completion chains met ekphrasis

The pattern was first discovered investigating ways of showing prompt/completion chains with ekphrasis. The complexity of TILTH gave way to the simplicity of the anecdote:

The Jar Based on Wallace Stevens's "Anecdote of a Jar" — the aha moment
I placed a jar in Tennessee,
And round it was, upon a hill.
It made the slovenly wilderness
Surround that hill.

The jar is the void. It doesn't fill the wilderness—it organizes it. This became the foundation for void management: the LLM manages a bounded space, not an unbounded world.

The Tetrad Operators

McLuhan's tetrad—Enhance, Obsolesce, Retrieve, Reverse—became the D-pad of a speculative design game. What if these four laws of media were operationalized as a prompt controller?

The Technical Uncanny Valley

Over a hundred test scenes built. API costs mounted. Found myself in a space not technical enough for computer scientists (Parag Mital at NeurIPS, emergentic.ai's multi-agent architectures) but too technical for design collaborators. Didn't meet the threshold yet—but learned enormously about software engineering, simulation architecture, and most importantly: context engineering.

The Key Realization: Void Management

The Shift Using LLM to construct scenes led to frustration with fine-grain controls. Assembly of the scene was the issue. This was resolved by seeing places where LLMs got in the way. The shift: LLM managing a void instead of a scene. The LLM becomes a gatekeeper, not a world-builder.

Problems Encountered

  • Context window degradation — scene graphs and ring buffer memory needed
  • Export problem — how to share something generated with LLM collaboration that isn't just a "vibe coded" website or raw text? (ONYX Studio attempts to address this)
  • Sunken cost fallacy — total overhaul of system would be costly, API bleeding

The Path Forward

These collaborations pushed toward novel 3D scenes—more clearly representing scenarios. This led to HOLO Project and eventually TRACTOR-DCE-GYO: a major pivot, creating a whole suite of tools, learning linear algebra, shifting from LEGOS-multi-channel scene graphs to richer disc-data.json formats.

HOLO Project Embedding narrative camera movement into prebaked format—films/scenarios shareable without LLM access
